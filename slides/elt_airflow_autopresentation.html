Fundamentos

¿Por qué ELT y no ETL?

ETL clásico

Transforma antes de cargar. Requiere un motor intermedio. Costoso en latencia y mantenimiento.

ELT moderno

Carga primero, transforma dentro del DWH. Aprovecha BigQuery, Snowflake o Redshift a escala.

En ELT, el data warehouse es el motor de transformación. Airflow solo orquesta cuándo y en qué orden ocurre cada paso.
03 / 12
Arquitectura

Componentes clave de Airflow

1
DAG — Grafo dirigido acíclico. Define el pipeline completo como código Python.
2
Operator — Unidad de trabajo: PythonOperator, PostgresOperator, BashOperator…
3
Hook — Conexión reutilizable a servicios externos (PostgresHook, S3Hook, HttpHook).
4
XCom — Pasa datos pequeños (IDs, conteos) entre tareas del mismo DAG.
5
Scheduler — Motor que evalúa y lanza los DAGs según su schedule.

Estructura DAG

Anatomía de un DAG

from airflow import DAG from airflow.operators.python import PythonOperator from datetime import datetime, timedelta default_args = { "owner": "data-team", "retries": 3, "retry_delay": timedelta(minutes=5), "email_on_failure": True, } with DAG( dag_id="elt_pipeline_demo", default_args=default_args, schedule="0 6 * * *", # Cada día a las 6 AM start_date=datetime(2024, 1, 1), catchup=False, tags=["elt", "postgres"], ) as dag: pass # Las tareas se definen aquí

Paso 1 – Extract

Extracción desde una API REST

import requests, json def extract_from_api(**context): url = "https://api.ejemplo.com/v1/orders" headers = {"Authorization": "Bearer {{ var.value.API_TOKEN }}"} response = requests.get(url, headers=headers, timeout=30) response.raise_for_status() data = response.json() # Guardar en archivo temporal o XCom (datos pequeños) context["ti"].xcom_push(key="record_count", value=len(data)) with open("/tmp/orders_raw.json", "w") as f: json.dump(data, f) return len(data) extract_task = PythonOperator( task_id="extract_orders", python_callable=extract_from_api, )
07 / 12
Paso 2 – Load

Carga a tabla staging en PostgreSQL

from airflow.providers.postgres.hooks.postgres import PostgresHook import pandas as pd, json def load_to_staging(**context): with open("/tmp/orders_raw.json") as f: data = json.load(f) df = pd.DataFrame(data) df["_loaded_at"] = pd.Timestamp.now() # audit column hook = PostgresHook(postgres_conn_id="postgres_dwh") engine = hook.get_sqlalchemy_engine() # Carga en staging SIN transformar — schema on load df.to_sql("stg_orders", engine, schema="staging", if_exists="replace", index=False) print(f"Loaded {len(df)} rows into staging.stg_orders") load_task = PythonOperator( task_id="load_to_staging", python_callable=load_to_staging, )
08 / 12
Paso 3 – Transform

Transformación con SQL dentro del DWH

from airflow.providers.postgres.operators.postgres import PostgresOperator transform_task = PostgresOperator( task_id="transform_orders", postgres_conn_id="postgres_dwh", sql=""" INSERT INTO analytics.fact_orders SELECT order_id::BIGINT, customer_id::INT, UPPER(TRIM(status)) AS status, amount::NUMERIC(12,2), created_at::TIMESTAMPTZ AS order_date, DATE_TRUNC('month', created_at) AS order_month, _loaded_at FROM staging.stg_orders ON CONFLICT (order_id) DO UPDATE SET status = EXCLUDED.status, amount = EXCLUDED.amount, _loaded_at = EXCLUDED._loaded_at; """, )

Dependencias

Encadenar tareas en el DAG

from airflow.operators.empty import EmptyOperator start = EmptyOperator(task_id="start") end = EmptyOperator(task_id="end") # Data quality check después de transformar def check_quality(**context): hook = PostgresHook(postgres_conn_id="postgres_dwh") count = hook.get_first("SELECT COUNT(*) FROM analytics.fact_orders")[0] if count == 0: raise ValueError("Quality check failed: 0 rows in fact_orders") quality_task = PythonOperator( task_id="quality_check", python_callable=check_quality, ) # Definir el flujo completo: start >> extract_task >> load_task >> transform_task >> quality_task >> end
El operador >> define dependencias. Airflow ejecuta las tareas respetando este orden.
10 / 12
Resiliencia

Reintentos y manejo de errores

Retries automáticos

Define retries=3 y retry_delay en default_args. Airflow reintenta la tarea fallida sin intervención.

Alertas por email

Activa email_on_failure=True y configura el SMTP en airflow.cfg para recibir notificaciones.

SLA miss

Define un SLA por tarea. Si no termina en el tiempo esperado, Airflow dispara un callback personalizado.

Callbacks

Usa on_failure_callback para enviar alertas a Slack, PagerDuty o cualquier endpoint.

Buenas prácticas

Principios de un pipeline production-ready

✓
Idempotencia: cada ejecución del pipeline debe producir el mismo resultado si los datos no cambian.
✓
No uses XCom para grandes volúmenes — usa S3, GCS o una tabla temporal como intermediario.
✓
Guarda credenciales en Connections — nunca hardcodees passwords en el código del DAG.
✓
Catchup=False en producción salvo que necesites backfill histórico explícito.
✓
Loguea métricas — registra filas procesadas, duración y errores en cada tarea para auditoría.
