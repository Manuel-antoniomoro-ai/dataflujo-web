Regla de oro: hechos tienen números, dimensiones tienen descriptores.

importe, coste, cantidad + 4 claves foráneas. Granularidad: 1 fila = 1 línea de venta.

-- Tabla de hechos
CREATE TABLE fact_ventas (
  venta_id      INT PRIMARY KEY,

  mes          TINYINT,
  mes_nombre   VARCHAR(20),

05 · DAX

Tabla de fechas
con DAX CALENDAR

Obligatoria para las funciones de inteligencia temporal. Sin ella, SAMEPERIODLASTYEAR devolverá error.

⚠️ Paso crítico: Tras crear la tabla, haz clic derecho → "Marcar como tabla de fechas" y selecciona la columna fecha.
📅 Rango: La tabla debe cubrir todas las fechas de tus datos, sin huecos. CALENDAR genera una fila por cada día del rango.
Nueva tabla — dim_tiempo (DAX)
1dim_tiempo =
2ADDCOLUMNS(
3  CALENDAR(DATE(2020,1,1), DATE(2025,12,31)),
4  "fecha_id", FORMAT([Date],"YYYYMMDD")*1,
5  "año", YEAR([Date]),
6  "trimestre", "Q" & QUARTER([Date]),
7  "mes", MONTH([Date]),
8  "mes_nombre", FORMAT([Date],"MMMM"),
9  "mes_año", FORMAT([Date],"MMM YYYY"),
10  "semana", WEEKNUM([Date]),
11  "dia_semana", FORMAT([Date],"DDDD"),
12  "es_fin_semana", IF(WEEKDAY([Date],2)>=6,1,0)
13)
06 · DAX

Medidas base
del P&L

Construye primero las medidas simples. La complejidad va en las medidas de nivel superior que las usan.

🗂️ Buena práctica: Crea una tabla "Medidas" vacía y pon todas las medidas ahí. Nunca mezcles medidas con columnas calculadas.
📐 Formato: Formatea cada medida al crear. Ingresos → Moneda. Margen % → Porcentaje. Ahorra problemas después.
❌ Error común: Usar SUM directamente en un visual sin encapsularlo en una medida. Pierde el contexto de filtro del RLS.
Medidas — Nueva medida (DAX)
1-- ① Medida base: Ingresos
2Ingresos = SUM(fact_ventas[importe])

5-- ② Coste Total
6Coste Total = SUM(fact_ventas[coste])

9-- ③ Margen Bruto (medida compuesta)
10Margen Bruto = [Ingresos] - [Coste Total]

13-- ④ Margen % con DIVIDE seguro
14Margen % =
15  DIVIDE([Margen Bruto], [Ingresos], 0)

18-- ⑤ Cantidad
19Cantidad Total = SUM(fact_ventas[cantidad])

22-- ⑥ Ticket medio
23Ticket Medio =
24  DIVIDE([Ingresos], [Cantidad Total], 0)
07 · DAX

Inteligencia temporal
YoY · MoM · YTD

LY
Ingresos Last Year
SAMEPERIODLASTYEAR desplaza el contexto de fecha exactamente un año hacia atrás.
YoY
Variación Year over Year %
Diferencia entre año actual y año anterior, expresada en porcentaje. La favorita de los CEOs.
YTD
Ingresos Year to Date
Acumulado desde el 1 de enero hasta la fecha seleccionada en el filtro.
LY
YTD
Ingresos LY YTD
Acumulado del año anterior hasta la misma fecha. Para comparar dónde estabas.
⚠️ Recuerda: Estas funciones requieren que dim_tiempo esté marcada como tabla de fechas. Si no, devolverán BLANK.
Medidas de inteligencia temporal
1-- Year over Year
2Ingresos LY =
3  CALCULATE(
4    [Ingresos],
5    SAMEPERIODLASTYEAR(dim_tiempo[fecha])
6  )

8YoY % =
9  DIVIDE([Ingresos] - [Ingresos LY], [Ingresos LY], 0)

12-- Year to Date
13Ingresos YTD =
14  CALCULATE([Ingresos], DATESYTD(dim_tiempo[fecha]))

17-- LY YTD para comparativa
18Ingresos LY YTD =
19  CALCULATE(
20    [Ingresos],

10 · DirectQuery

Configurar
DirectQuery

DirectQuery = datos siempre frescos. El precio: rendimiento dependiente de SQL Server.

01
Seleccionar modo al conectar
En el diálogo de conexión a SQL Server, elige "DirectQuery" (no Importar).
02
Límite de tiempo de consulta
Opciones → DirectQuery → timeout 30s. Evita que queries lentas bloqueen el dashboard.
03
Activar log de consultas
Opciones → Diagnóstico → Activar registro. Esencial para depurar rendimiento.
04
Query Reduction
Opciones → DirectQuery → "Reducir número de consultas". Agrupa peticiones.
❌ No puedes: Usar columnas calculadas en DirectQuery. Todo cálculo en columna debe hacerse en SQL Server (vistas o columnas calculadas de SQL).
SQL generado por Power BI (log)
-- Query generada automáticamente por PBI
-- al filtrar por Q1 2025, Dept = Ventas
SELECT
  d.[mes_nombre],
  SUM(f.[importe]) AS [Ingresos],
  SUM(f.[coste]) AS [Coste Total]
FROM [dbo].[fact_ventas] f
JOIN [dbo].[dim_tiempo] d
  ON f.[fecha_id] = d.[fecha_id]
JOIN [dbo].[dim_departamento] dp
  ON f.[dept_id] = dp.[dept_id]
WHERE
  d.[año] = 2025
  AND d.[trimestre] = 1
  AND dp.[director_email] = 'dir.ventas@empresa.es'
GROUP BY d.[mes_nombre]
ORDER BY d.[mes]
👁️ Observa: Power BI inyecta automáticamente el filtro del RLS (director_email = '...') en cada query. El usuario nunca puede ver datos de otro departamento.
11 · SQL Server

Optimización
para DirectQuery

IDX
Índices en columnas de filtro
Índice compuesto en (fecha_id, dept_id, cliente_id) en fact_ventas. Reduce tiempo de query 10x.
VIEW
Conectar a vistas, no a tablas
Las vistas encapsulan lógica y permiten que el DBA optimice sin tocar Power BI.
STAT
Estadísticas actualizadas
UPDATE STATISTICS semanalmente. Las estadísticas obsoletas generan planes de ejecución malos.
PART
Particionado por fecha
Si la tabla supera 10M de filas, particiona por fecha_id. Reduce scans a la partición del período.
-- Índice compuesto fact_ventas
CREATE NONCLUSTERED INDEX
  [IX_fact_ventas_filtros]
ON [dbo].[fact_ventas] (
  [fecha_id] ASC,
  [dept_id] ASC,
  [cliente_id] ASC
)
INCLUDE ([importe], [coste], [cantidad]);

-- Vista para Power BI (encapsula lógica)
CREATE VIEW [dbo].[vw_ventas_bi] AS
SELECT
  f.venta_id, f.importe, f.coste,
  t.año, t.trimestre, t.mes_nombre,
  c.nombre AS cliente,
  c.segmento,
  d.nombre AS departamento,
  d.director_email
FROM fact_ventas f
JOIN dim_tiempo t ON f.fecha_id = t.fecha_id
JOIN dim_cliente c ON f.cliente_id = c.cliente_id
JOIN dim_departamento d ON f.dept_id = d.dept_id;

-- Actualizar estadísticas
UPDATE STATISTICS [dbo].[fact_ventas]
  WITH FULLSCAN;

19 · Debugging

6 errores comunes — y cómo evitarlos

❌ 1. Relaciones bidireccionales

Activan ambas direcciones de filtro y generan resultados incorrectos. Usa unidireccional siempre. Bidireccional solo en casos muy específicos documentados.
❌ 2. No marcar tabla de fechas

Sin marcar dim_tiempo como tabla de fechas, todas las funciones de inteligencia temporal fallan silenciosamente devolviendo BLANK.
❌ 3. CALCULATE sin modificador

CALCULATE([Ingresos]) = [Ingresos]. Sin función de contexto (FILTER, ALL, SAMEPERIODLASTYEAR...) CALCULATE no hace nada.
❌ 4. Columnas calculadas en vez de medidas

Las columnas calculadas consumen RAM para cada fila. En DirectQuery no existen. Usa medidas siempre que sea posible.
❌ 5. No gestionar BLANK

Una medida BLANK rompe totales y gráficos. Usa COALESCE([Medida], 0) o IF(ISBLANK([M]), 0, [M]).
❌ 6. Publicar sin probar RLS

Prueba cada rol con "Ver como" antes de publicar. Un RLS incorrecto puede exponer datos confidenciales a usuarios no autorizados.

21 · Optimización

DAX con VAR:
más rápido y legible

VAR evalúa la expresión una sola vez y la reutiliza. Menor tiempo de evaluación y código más limpio.

⚡ Rendimiento: Sin VAR, cada referencia a [Ingresos LY] lanza una evaluación de CALCULATE. Con VAR, se evalúa una vez y se cachea.
📖 Legibilidad: VAR + RETURN hace el DAX auto-documentado. Cualquier analista puede auditarlo sin explicación.
🐛 Debugging: Puedes cambiar el RETURN para devolver cualquier VAR intermedia y ver su valor. Invaluable para depurar.
❌ Sin VAR (más lento)
YoY % =
  DIVIDE(
    [Ingresos] - CALCULATE([Ingresos],
