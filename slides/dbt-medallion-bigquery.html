00 · dbt Core

¿Qué es dbt?

La herramienta que transformó el data engineering en los últimos cinco años.

01
SQL-first
Cada modelo es un fichero .sql con una SELECT. Sin stored procedures ni ETL propietario.
02
Git versionado
Todo el código vive en Git. Pull requests, branches, historial completo.
03
Tests automáticos
not_null, unique, accepted_values, relationships out of the box.
04
Documentado
Catálogo automático con descripción de modelos y columnas.
05
DAG automático
dbt construye el grafo de dependencias y ejecuta en orden.
dbt Core 1.8 — Unit tests nativos y contratos de datos (data contracts) para validar esquemas entre modelos upstream y downstream.
-- models/staging/stg_orders.sql
{{ config(materialized='view') }}

with source as (
    select * from {{ source('raw', 'orders') }}
),
renamed as (
    select
        order_id,
        customer_id,
        cast(order_date as date)    as order_date,
        cast(amount as numeric)     as amount_eur,
        lower(trim(status))         as status,
        current_timestamp()        as _loaded_at
    from source
    where order_id is not null
)
select * from renamed

02 · Setup

Setup inicial del proyecto

01
pip install dbt-bigquery
Instala dbt-bigquery==1.8.*. Requiere Python 3.8+. Recomendado usar un entorno virtual.
02
dbt init mi_proyecto
Crea la estructura de carpetas y el dbt_project.yml inicial.
03
Configurar profiles.yml
Credenciales de BigQuery en ~/.dbt/profiles.yml. Nunca en Git.
04
dbt debug
Verifica la conexión a BigQuery. Debe mostrar All checks passed.
Consejo — Ejecuta siempre dbt debug antes de empezar a desarrollar. Detecta problemas de credenciales o red antes de perder tiempo.

01
Crear Service Account
IAM > Service Accounts en GCP. Roles: BigQuery Data Editor + BigQuery Job User.

Asignar roles IAM
BigQuery Data Editor para leer/escribir datasets. BigQuery Job User para ejecutar queries.

04 · Bronze

Capa Bronze: Staging Models

Primera capa de calidad. Datos tipados y normalizados desde el origen.

01
Renombrar columnas
Nombres snake_case consistentes. Sin abreviaciones crípticas. customer_id, no cust_id.
02
Castear tipos
Fechas como DATE o TIMESTAMP. Montos como NUMERIC. IDs como STRING.
03
Filtrar erróneos
WHERE order_id IS NOT NULL. Solo filtros evidentes, no lógica de negocio.
Regla de oro — Los modelos de staging NUNCA hacen JOINs. Si necesitas un JOIN, pertenece a la capa Silver.
Materialización — Siempre VIEW en staging. No duplicar datos del origen. BigQuery evalúa la vista en cada consulta.
-- models/staging/stg_orders.sql
{{ config(materialized='view') }}

with source as (
    select *
    from {{ source('raw', 'orders') }}
),
staged as (
    select
        order_id,
        customer_id,
        cast(order_date as date)      as order_date,
        cast(amount as numeric)       as amount_eur,
        lower(trim(status))           as status,
        current_timestamp()          as _loaded_at
    from source
    where order_id is not null
        -- Solo filtros de calidad básica
)
select * from staged
05 · Silver

Capa Silver: Intermediate Models

01
JOINs entre entidades
Aquí se unen stg_orders con stg_customers y stg_products.
02
Lógica de negocio
Derivar campos calculados, estandarizar categorías, gestionar nulos.
03
Campos calculados
margin_pct, fiscal_quarter, customer_lifetime_value.
04
Estandarizar categorías
Unificar valores equivalentes. 'COMPLETADO' y 'completed' son lo mismo.
05
Gestionar nulos
COALESCE para nulos esperados. Documentar los inesperados.
Regla del millón — Si el modelo supera 1M filas o lo usan más de 3 modelos downstream, materialízalo como TABLE. Evita recalcular lógica costosa en cada consulta.
-- models/intermediate/int_orders_enriched.sql
{{ config(materialized='table') }}

with orders as (
    select * from {{ ref('stg_orders') }}
),
customers as (
    select * from {{ ref('stg_customers') }}
),
products as (
    select * from {{ ref('stg_products') }}
),
enriched as (
    select
        o.order_id, o.order_date, o.amount_eur, o.status,
        c.customer_name, c.country, c.segment,
        p.category, p.brand
    from orders o
    left join customers c using (customer_id)
    left join products p using (product_id)
)
select * from enriched
06 · Gold

Capa Gold: Mart Models

01
marts/finance
mart_revenue, mart_pl, mart_cashflow. Owner: equipo Finance.
02
marts/sales
mart_sales_pipeline, mart_quota_attainment. Owner: equipo Ventas.
03
marts/operations
mart_fulfillment, mart_inventory. Owner: equipo Operaciones.
Denormalización inteligente — En los marts está bien repetir columnas descriptivas. Los dashboards no deben hacer JOINs. Si el analista necesita un JOIN en BI, esa métrica pertenece al mart.
-- models/marts/finance/mart_revenue.sql
{{ config(materialized='table') }}

with base as (
    select * from {{ ref('int_orders_enriched') }}
    where status = 'completed'
),
aggregated as (
    select
        order_date,
        country,
        segment,
        category,
        count(distinct order_id) as order_count,
        sum(amount_eur)            as revenue_eur,
        avg(amount_eur)            as avg_order_eur
    from base
    group by 1, 2, 3, 4
)
select * from aggregated
07 · Sources

Sources y ref()

Las dos macros fundamentales de dbt para referenciar datos.

# models/staging/sources.yml
version: 2
sources:
  - name: raw
    database: mi-proyecto-gcp
    schema: raw_data
    freshness:
      warn_after: {count: 6, period: hour}
      error_after: {count: 12, period: hour}
    tables:
      - name: orders
        loaded_at_field: _created_at
      - name: customers
      - name: products
Regla única — Si la tabla viene de fuera de dbt, usa source(). Si viene de otro modelo dbt, usa ref(). Nunca mezcles los dos para el mismo objeto.
-- ✅ Source: tabla EXTERNA a dbt
select * from {{ source('raw', 'orders') }}
-- Resuelve a: mi-proyecto.raw_data.orders

-- ✅ Ref: modelo dbt INTERNO
select * from {{ ref('stg_orders') }}
-- Dev: mi-proyecto.dbt_dev.stg_orders
-- Prod: mi-proyecto.dbt_prod.stg_orders

-- ❌ NUNCA source() para modelos dbt
-- {{ source('staging', 'stg_orders') }}
-- Rompe el DAG: dbt no ve la dependencia

-- Chequear frescura de sources:
-- $ dbt source freshness
08 · Tests

Tests genéricos de dbt

01
not_null
Ningún valor puede ser NULL. Obligatorio en todas las claves primarias.
02
unique
Los valores son únicos en la columna. Obligatorio en claves primarias.
03
accepted_values
Los valores pertenecen a un conjunto definido. Para status, tipo, país.
04
relationships
La clave foránea existe en la tabla referenciada. Integridad referencial.
dbt build — Ejecuta dbt run + dbt test en un solo comando, en orden de dependencias. Siempre usa dbt build en CI/CD.
# models/staging/schema.yml
version: 2
models:
  - name: stg_orders
    description: Pedidos normalizados
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: status
        tests:
          - accepted_values:
              values: ['completed', 'pending', 'cancelled']
      - name: customer_id
        tests:
          - not_null
          - relationships:
              to: ref('stg_customers')
              field: customer_id
09 · Tests Custom

Tests custom con macros Jinja

01
Lógica de negocio custom
Validar que el total de factura cuadra con la suma de líneas.
02
SQL devuelve filas fallidas
El test falla si la query devuelve CUALQUIER fila. Filas = errores.
03
Vacío = OK, Filas = FAIL
Convención universal de dbt para todos los tests.
04
Misma sintaxis en schema.yml
Los tests custom se configuran igual que not_null o unique.
Reutilización — Los tests en tests/generic/ se usan en cualquier modelo del proyecto. Escríbelos una vez, aplícalos en todos los schema.yml.
-- tests/generic/assert_positive_amount.sql
{% test assert_positive_amount(model, column_name) %}

with validation as (
    select
        {{ column_name }} as valor,
        count(*) as n_failures
    from {{ model }}
    where {{ column_name }} is null
        or {{ column_name }} < 0
    group by 1
)
select * from validation

{% endtest %}

-- Uso en schema.yml:
-- tests:
-- - assert_positive_amount:
-- column_name: amount_eur
10b · Unit Tests

Unit Tests nativos en dbt 1.8

Prueba la lógica SQL con datos mock en YAML. Sin conexión a BigQuery necesaria.

01
Fixtures de entrada en YAML
Filas mock definidas en YAML. Sin datos reales en BigQuery. Ciclo de dev mucho más rápido.
02
Outputs esperados
Define exactamente qué filas y columnas debe producir el modelo con esos fixtures.
03
dbt test --select test_type:unit
Ejecuta solo unit tests. Sin materializar modelos ni procesar datos reales de producción.
04
Falla si hay diferencias
Cualquier columna que no coincida con el expected genera un FAIL con detalle preciso.
Dos capas de testing — Unit tests prueban la LÓGICA (modelos). Tests de datos prueban la CALIDAD (producción). Son complementarios: ambos son necesarios en proyectos robustos.
# tests/unit/test_fiscal_quarter.yml
unit_tests:
  - name: test_fiscal_quarter_logic
    model: mart_revenue
    given:
      - input: ref('int_orders_enriched')
        rows:
          - {order_date: '2024-04-01', amount_eur: 1000, status: 'completed'}
          - {order_date: '2024-07-01', amount_eur: 2000, status: 'completed'}
          - {order_date: '2024-01-15', amount_eur: 500,  status: 'cancelled'}
    expect:
      rows:
        - {fiscal_q: 1, revenue_eur: 1000}
        - {fiscal_q: 2, revenue_eur: 2000}

# Ejecutar solo unit tests:
$ dbt test --select test_type:unit
# cancelled excluido: WHERE status = 'completed'
# ✅ 2 passed — lógica del trimestre verificada
10 · Macros

Macros Jinja reutilizables

01
get_fiscal_quarter
Calcula el trimestre fiscal configurable. Inicio de año en cualquier mes.
02
generate_surrogate_key
Clave subrogada MD5 de múltiples columnas. Para tablas sin PK natural.
03
safe_divide
División segura con NULLIF. Evita division by zero en métricas de ratio.
04
cents_to_euros
Conversión de céntimos a euros con precisión NUMERIC. Para sistemas legados.
Librería de macros — Mantén un repo de macros estándar para todos tus proyectos. packages.yml permite instalar paquetes de macros de la comunidad dbt (dbt_utils, dbt_date).
-- macros/get_fiscal_quarter.sql
{% macro get_fiscal_quarter(date_col, start_month=4) %}
    case
        when extract(month from {{ date_col }}) >= {{ start_month }}
        then cast(
            ceil((extract(month from {{ date_col }})
                - {{ start_month }} + 1.0) / 3)
        as int64)
        else cast(
            ceil((extract(month from {{ date_col }})
                + 12 - {{ start_month }} + 1.0) / 3)
        as int64)
    end
{% endmacro %}

-- Uso en mart_revenue.sql:
select order_id,
  {{ get_fiscal_quarter('order_date') }} as fiscal_q
from {{ ref('int_orders_enriched') }}
11 · Seeds

Seeds para tablas de catálogo

01
Versionado en Git
El CSV vive en el repo junto al SQL. Historial completo de cambios.
02
dbt seed
Carga el CSV en BigQuery como tabla. Primera vez crea, siguientes actualiza.
03
dbt seed --full-refresh
Recrea la tabla desde cero. Útil si cambia el esquema del CSV.
04
Referenciar con ref()
Los modelos usan ref('exchange_rates') igual que cualquier modelo.
Casos de uso ideales — Tablas de divisas, calendarios fiscales, mapas de categorías, listas de países ISO, configuración de umbrales de negocio.
Límite de tamaño — Para ficheros de más de 100K filas usa una tabla BigQuery con carga incremental, no un seed. dbt seed no está optimizado para volúmenes grandes.

12b · Packages

Packages y dbt_utils

El ecosistema de packages añade macros probadas sin escribir código desde cero.

01
packages.yml + dbt deps
Declara packages con versión fija. dbt deps los instala en dbt_packages/. Nunca a Git.
02
generate_surrogate_key
MD5 de múltiples columnas. Clave subrogada cuando no hay PK natural en la fact table.
03
date_spine
Genera secuencia continua de fechas. Detecta huecos en series temporales sin gaps.
04
audit_helper
Compara dos versiones de un modelo. Detecta diferencias antes de desplegar a producción.
dbt_packages/ en .gitignore — Nunca subas dbt_packages/ al repo. Se genera con dbt deps, igual que node_modules. Añade dbt_packages/ y target/ al .gitignore.
# packages.yml (raíz del proyecto)
packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.2.0", "<2.0.0"]
  - package: calogica/dbt_date
    version: [">=0.10.0", "<1.0.0"]
  - package: dbt-labs/audit_helper
    version: [">=0.9.0", "<1.0.0"]

# Instalar: $ dbt deps

-- Uso de dbt_utils en mart_revenue.sql
select
    {{ dbt_utils.generate_surrogate_key([
        'order_id', 'order_date'

13 · Incremental

Modelos incrementales

01
materialized='incremental'
Solo procesa datos nuevos desde la última ejecución. Ideal para fact tables.
02
unique_key
Columna(s) que identifican una fila de forma única. Se usa para el MERGE.
03
is_incremental()
Macro que activa el filtro de delta solo en ejecuciones no-full-refresh.
04
--full-refresh
Fuerza la reconstrucción completa. Ejecutar una vez al mes en producción.
Primera ejecución — La primera vez que ejecutas un modelo incremental, dbt carga TODOS los datos (full refresh automático). Las siguientes solo el delta. BigQuery MERGE es transaccional.
-- models/marts/fact_events_incremental.sql
{{ config(
    materialized     = 'incremental',
    unique_key       = 'event_id',
    on_schema_change = 'sync_all_columns'
) }}

select
    event_id,
    user_id,
    event_type,
    cast(event_timestamp as timestamp) as event_ts,
    session_id,
    properties
from {{ source('raw', 'events') }}

{% if is_incremental() %}
where event_timestamp > (
    select max(event_ts) from {{ this }}
)
{% endif %}
14b · Snapshots

Snapshots: historial SCD Type 2

Captura el historial de cambios en tablas que se sobreescriben. SCD2 con dbt.

01
dbt snapshot
Ejecuta todas las snapshots. Primera vez crea la tabla. Siguientes añaden historial.
02
Estrategia timestamp
Compara el campo updated_at. Eficiente para tablas con timestamp de modificación.
03
Estrategia check
Compara columnas específicas. Funciona sin campo updated_at en la tabla fuente.
04
dbt_valid_from / dbt_valid_to
Metadatos generados automáticamente. dbt_valid_to = NULL indica el registro activo.
⚠️ Frecuencia de ejecución — Las snapshots capturan el estado en el momento de correr. Si no ejecutas dbt snapshot durante 3 días, perderás los cambios intermedios. Ejecútalas mínimo una vez al día.
-- snapshots/customers_snapshot.sql
{% snapshot customers_snapshot %}

{{ config(
    target_schema = 'snapshots',
    unique_key    = 'customer_id',
    strategy      = 'timestamp',
    updated_at    = 'updated_at'
) }}

select * from {{ source('raw', 'customers') }}

{% endsnapshot %}

-- Resultado en snapshots.customers_snapshot:
-- customer_id | segment | dbt_valid_from | dbt_valid_to
-- 101         | small | 2024-01-01 | 2024-06-15
-- 101         | enterprise | 2024-06-15 | NULL ← activo
14c · Variables

Variables y env_var()

Parametriza modelos por entorno. Misma lógica, distinto comportamiento en dev vs prod.

01
var() en dbt_project.yml
Variables con valores por defecto. Sobreescribibles con --vars en CLI o CI/CD.
02
env_var()
Variables de entorno del SO. Para secrets y configuración sensible. Nunca a Git.
03
target.name
Nombre del entorno activo: dev / staging / prod. Para condicionar lógica por entorno.
04
Coste reducido en dev
Filtra los últimos 30 días en dev. Misma lógica, 95% menos datos escaneados en BigQuery.
Segundo parámetro de env_var() — env_var acepta un valor por defecto: env_var('SLACK_WEBHOOK', ''). Si la variable no existe, usa el default en vez de lanzar un error. Muy útil para desarrollo local.
# dbt_project.yml
vars:
  start_date: '2020-01-01'    # override: --vars
  enable_new_feature: false
  slack_url: {{ env_var('SLACK_WEBHOOK', '') }}

-- models/marts/mart_revenue.sql
select * from {{ ref('int_orders_enriched') }}
where order_date >= '{{ var("start_date") }}'

-- En dev: procesar solo últimos 30 días
{% if target.name == 'dev' %}
  and order_date >= date_sub(current_date(), interval 30 day)
{% endif %}

# CLI: sobreescribir variable
$ dbt run --vars '{"start_date": "2024-01-01"}'
14d · Selectors

Tags y selectors avanzados

Ejecuta subconjuntos del pipeline con precisión. Imprescindible en proyectos grandes.

01
Tags por capa en dbt_project.yml
+tags: ['daily', 'finance'] en staging, intermediate, marts. Agrupa por frecuencia y dominio.
02
+modelo (upstream)
Ejecuta el modelo más todos sus ancestors. Para reconstruir desde la fuente.
03
modelo+ (downstream)
Ejecuta el modelo más todos los que dependen de él. Para propagar un cambio.
04
state:modified+
Solo modelos modificados en el último commit más su downstream. CI/CD incremental.
CI/CD incremental — dbt run --select state:modified+ en el pipeline de PR ejecuta solo los modelos afectados por el cambio. Reduce el tiempo de CI de 15 minutos a 2 minutos en proyectos grandes.
# dbt_project.yml — tags por capa
models:
  mi_proyecto:
    staging:
      +tags: ['staging', 'bronze', 'daily']
    intermediate:
      +tags: ['silver', 'daily']
    marts:
      finance:
        +tags: ['gold', 'finance', 'daily']
      operations:
        +tags: ['gold', 'ops', 'weekly']

# CLI — selectors clave:
$ dbt run --select tag:daily
$ dbt run --select +mart_revenue            # + upstream
$ dbt run --select mart_revenue+           # downstream +
$ dbt run --select state:modified+         # solo cambios
$ dbt test --select staging               # tests de capa
$ dbt run --exclude tag:weekly             # excluir
14 · Optimización

Optimización en BigQuery

01
partition_by
Segmentos físicos por fecha. Solo escanea las particiones del rango filtrado.
02
cluster_by
Ordena físicamente por columnas de filtro frecuente. Reduce scan dentro de la partición.
03
BigQuery cobra por scan
5 USD por TB. Particionado + clustering reduce 80-95% el scan en tablas grandes.
04
Cuándo aplicar
Siempre en modelos mart con más de 100M filas. Impacto directo en factura GCP.
Ahorro real — En tablas de 1B filas, partition_by + cluster_by puede reducir el coste de consulta de 50 USD a menos de 1 USD por ejecución. ROI inmediato en producción.
-- models/marts/finance/mart_revenue_optimized.sql
{{ config(
    materialized = 'table',
    partition_by = {
        'field'       : 'order_date',
        'data_type'   : 'date',
        'granularity' : 'day'
    },
    cluster_by = ['country', 'segment', 'category'],
    labels = {'env': 'prod', 'team': 'finance'}
) }}

select
    order_date, country, segment, category,
    sum(amount_eur)          as revenue_eur,
    count(distinct order_id) as order_count
from {{ ref('int_orders_enriched') }}
where status = 'completed'
group by 1, 2, 3, 4

Airflow + dbt en producción

Orquesta dbt con Airflow para pipelines robustos con alertas y reintentos automáticos.

01
BashOperator básico
dbt run + dbt test en tareas secuenciales. Setup en minutos. Ideal para proyectos simples.
02
astronomer-cosmos
Cada modelo dbt = tarea Airflow individual. Granularidad total. Retry por modelo.
03
dbt source freshness primero
Verifica que los datos origen están actualizados antes de materializar. Falla rápido.
04
Retries con backoff
retries=2, retry_delay=5min. Para errores transitorios de red o BigQuery API.
Airflow vs dbt Cloud — dbt Cloud tiene su propio scheduler. Si ya tienes Airflow en tu stack, úsalo para dbt también. Centraliza la orquestación y el monitoring en un solo lugar.
# dags/dbt_medallion_pipeline.py
from airflow import DAG
from airflow.operators.bash import BashOperator
from datetime import datetime, timedelta

with DAG(
    dag_id     = 'dbt_medallion',
    start_date = datetime(2024, 1, 1),
    schedule   = '0 6 * * *',      # 6:00 cada día
    catchup    = False,
    default_args = {
        'retries': 2,
        'retry_delay': timedelta(minutes=5),
        'on_failure_callback': send_slack_alert
    }) as dag:

    freshness = BashOperator(task_id='source_freshness',
        bash_command='dbt source freshness --target prod')

    run = BashOperator(task_id='dbt_run',
        bash_command='dbt run --target prod --tag daily')

    test = BashOperator(task_id='dbt_test',
        bash_command='dbt test --target prod --tag daily')

    freshness >> run >> test

17 · Errores

5 errores comunes en dbt

01 Marts como VIEW
Los marts materializados como view hacen que el dashboard ejecute la query completa en cada carga. Siempre +materialized: table en marts.
02 source() para modelos dbt
Usar source() para referenciar un modelo dbt rompe el DAG. dbt no ve la dependencia. Siempre ref() para modelos dbt.
03 Ambiguous column
JOIN sin alias de tabla genera 'ambiguous column'. Siempre usa alias explícitos: o.order_id, c.customer_name.
04 Relation not found
Ejecutar dbt run --select mi_modelo sin sus deps. Usa dbt run --select +mi_modelo para incluir dependencias upstream.
05 dbt run sin dbt test
dbt run NO ejecuta los tests. Para materializar y testear en un paso: dbt build. Siempre dbt build en CI/CD.
Regla de oro — Usa siempre dbt build en lugar de dbt run. Materializa, testa y documenta en un solo comando. Si un test falla, el build se detiene antes de llegar a los marts.
