01 · Concepto

GROUP BY
vs OVER()

La diferencia fundamental que cambia cómo piensas en SQL.

❌ GROUP BY — colapsa filas
N filas → 1 fila por grupo
Pierdes el detalle de cada transacción. No puedes calcular % sobre total en la misma query.
✅ OVER() — ventana deslizante
N filas → N filas (más columnas calculadas)
Cada fila conserva su detalle más el valor calculado sobre el grupo. Puedes combinar métricas de fila y de grupo en la misma query.
💡 Regla práctica: Si necesitas un resultado por grupo → GROUP BY. Si necesitas el detalle de cada fila más información del grupo → Window Function.
❌ Con GROUP BY — pierdes el detalle
SELECT cliente_id,
  SUM(importe) AS total_cliente
FROM ventas
GROUP BY cliente_id
-- Resultado: 1 fila por cliente. Pierdes las ventas individuales.
✅ Con OVER() — conservas el detalle
SELECT
  venta_id, cliente_id, importe,
SUM(importe) OVER (PARTITION BY cliente_id)     AS total_cliente,
ROUND(importe * 100.0 / SUM(importe) OVER (PARTITION BY cliente_id), 1) AS pct_sobre_total FROM ventas
-- Resultado: todas las filas + total del cliente + % de cada venta
02 · Sintaxis

Anatomía de
la cláusula OVER()

PRT
PARTITION BY
Define los grupos (ventanas). Opcional. Sin él, toda la tabla es una ventana.
ORD
ORDER BY
Orden dentro de la partición. Obligatorio para ROW_NUMBER, RANK, LAG, LEAD y acumulados.
FRM
FRAME (ROWS / RANGE BETWEEN)
Define qué filas concretas entran en el cálculo. Controla acumulados y medias móviles.
⚠️ Bug silencioso: Sin frame explícito y con ORDER BY, el frame por defecto es RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, lo que puede dar resultados inesperados en LAST_VALUE.
Estructura completa
FUNCIÓN(columna_o_expresión)
OVER (
  PARTITION BY col1, col2   opcional — grupos
  ORDER BY col3 ASC/DESC  orden en partición
  ROWS BETWEEN inicio AND fin frame
)
Valores del FRAME
UNBOUNDED PRECEDING Desde la primera fila de la partición
N PRECEDING N filas antes de la fila actual
CURRENT ROW La fila actual
N FOLLOWING N filas después de la fila actual
UNBOUNDED FOLLOWING Hasta la última fila de la partición
03 · Ranking

ROW_NUMBER()
— Deduplicación

Numeración única por partición. El arma definitiva para deduplicar datos.

①
Asigna número único por partición
Sin empates. Dos filas iguales reciben números distintos. Determinista con ORDER BY claro.
②
Deduplicar → filtrar WHERE rn = 1
Ordena por fecha DESC. La fila más reciente obtiene rn=1. Filtra en CTE exterior.
③
Paginación eficiente
ROW_NUMBER() en vez de OFFSET/LIMIT es más rápido en tablas grandes en todos los motores.
⚡ Rendimiento: En BigQuery y Snowflake, ROW_NUMBER con PARTITION BY y ORDER BY sobre columnas con clustering o partición es extremadamente eficiente. Mucho más que DISTINCT o subqueries correlacionadas.
SQL — Deduplicar clientes
-- Problema: tabla clientes con duplicados históricos
-- Solución: quedarnos con el registro más reciente

WITH ranked AS (
  SELECT
    cliente_id, nombre, email,
    fecha_actualizacion,
ROW_NUMBER() OVER ( PARTITION BY cliente_id ORDER BY fecha_actualizacion DESC ) AS rn   FROM dim_clientes
)
SELECT cliente_id, nombre, email, fecha_actualizacion
FROM ranked
WHERE rn = 1  -- Solo la fila más reciente por cliente

04 · Ranking

RANK() vs
DENSE_RANK()

Tres funciones similares, comportamientos distintos ante empates.

ROW_NUMBER
Siempre único. Sin empates. 1, 2, 3, 4...
→ Deduplicar, paginar
RANK
Empates = mismo número + salta posiciones. 1, 1, 3, 4...
→ Rankings con huecos (deportes, concursos)
DENSE_RANK
Empates = mismo número sin saltos. 1, 1, 2, 3...
→ Top N sin importar empates
💡 Caso real: Top 5 productos por región sin subqueries. Usa DENSE_RANK y filtra WHERE dr <= 5.
SQL — Top 3 vendedores por región
WITH ranked AS (
  SELECT
    region, vendedor, ventas_total,
ROW_NUMBER() OVER ( PARTITION BY region ORDER BY ventas_total DESC ) AS rn, RANK() OVER ( PARTITION BY region ORDER BY ventas_total DESC ) AS rk, DENSE_RANK() OVER ( PARTITION BY region ORDER BY ventas_total DESC ) AS dr   FROM ventas_vendedor
)
SELECT * FROM ranked WHERE dr <= 3

05 · Desplazamiento

LAG()
— Período anterior

Accede al valor de N filas anteriores dentro de la partición. Imprescindible para MoM y YoY.

LAG(columna, offset, default)
OVER (PARTITION BY … ORDER BY …)
offset=1
Fila inmediatamente anterior
MoM: LAG(ingresos, 1, 0) → ingresos del mes pasado.
offset=12
12 filas atrás
YoY con datos mensuales: LAG(ingresos, 12, 0) → mismo mes del año anterior.
default
Valor cuando no hay fila anterior
Evita NULL en la primera fila. Usa 0 o NULL según el contexto.
⚠️ Cuidado: El ORDER BY del OVER debe ser el mismo que el orden lógico de tus períodos. Si tienes meses como texto "Ene", "Feb"… ordena por columna numérica de mes, no por nombre.
SQL — Variación MoM y YoY
SELECT
  año, mes, ingresos,

  -- Mes anterior (MoM)
LAG(ingresos, 1, 0) OVER ( PARTITION BY año ORDER BY mes ) AS ingresos_mes_anterior,
  -- Variación absoluta MoM
  ingresos - LAG(ingresos, 1, 0) OVER
    (PARTITION BY año ORDER BY mes)
  AS var_mom_abs,

  -- Variación % MoM
ROUND((ingresos - LAG(ingresos,1,0) OVER (PARTITION BY año ORDER BY mes)) * 100.0 / NULLIF(LAG(ingresos,1,0) OVER (PARTITION BY año ORDER BY mes),0), 1) AS var_mom_pct
FROM metricas_mensuales
ORDER BY año, mes
06 · Desplazamiento

LEAD()
— Detección de churn

Mira N filas hacia adelante. Detecta clientes que no volverán y calcula tiempo entre eventos.

LEAD(columna, offset, default)
OVER (PARTITION BY … ORDER BY …)
LEAD IS NULL
No hay compra siguiente → cliente churneado
→ Identificar la última compra antes del churn
días entre compras
LEAD(fecha) - fecha_actual = días hasta siguiente pedido
→ Frecuencia de compra por segmento
💡 Churn real: Combina LEAD con una ventana de tiempo. Un cliente sin compra en los siguientes 90 días = churned. Filtra WHERE días_siguiente_compra > 90 OR siguiente IS NULL.
SQL — Detección de churn
WITH compras_lead AS (
  SELECT
    cliente_id,
    fecha_compra,
    importe,
LEAD(fecha_compra) OVER ( PARTITION BY cliente_id ORDER BY fecha_compra ) AS siguiente_compra,
DATEDIFF(day, fecha_compra, LEAD(fecha_compra) OVER ( PARTITION BY cliente_id ORDER BY fecha_compra )) AS dias_hasta_siguiente
  FROM pedidos
)
SELECT
  cliente_id, fecha_compra, importe,
  siguiente_compra, dias_hasta_siguiente,
  CASE
WHEN siguiente_compra IS NULL THEN 'CHURNED'     WHEN dias_hasta_siguiente > 90 THEN 'EN RIESGO'
    ELSE 'ACTIVO'
  END AS estado_cliente
FROM compras_lead
07 · Agregados

SUM OVER()
— YTD y Running Total

Acumulados sin perder el detalle de cada fila. El frame determina qué filas entran en la suma.

UNBOUNDED PRECEDING → CURRENT ROW
Running total acumulado hasta la fila actual. Para YTD.
UNBOUNDED PRECEDING → UNBOUNDED FOLLOWING
Total completo de la partición. Igual en todas las filas. Para calcular %.
N PRECEDING → CURRENT ROW
Suma de los últimos N períodos. Para ventanas móviles.
⚠️ Sin frame explícito: Si tienes ORDER BY, el frame por defecto es UNBOUNDED PRECEDING TO CURRENT ROW (acumulado). Ponlo siempre explícito para que el código sea auto-documentado.
SQL — YTD + % sobre total anual
SELECT
  año, mes, mes_nombre, ingresos,

  -- YTD: acumulado desde enero hasta este mes
SUM(ingresos) OVER ( PARTITION BY año ORDER BY mes ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS ytd,
  -- Total del año completo (mismo valor en todas las filas)
SUM(ingresos) OVER ( PARTITION BY año ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING ) AS total_anual,
  -- Porcentaje sobre total anual
ROUND(ingresos * 100.0 / SUM(ingresos) OVER (PARTITION BY año ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING), 1) AS pct_anual
FROM metricas_mensuales ORDER BY año, mes
08 · Agregados

AVG OVER()
— Media móvil 7/30 días

Suaviza series temporales ruidosas. Elimina el efecto día-de-la-semana.

7d
Media móvil 7 días
ROWS BETWEEN 6 PRECEDING AND CURRENT ROW. Elimina ruido semanal.
30d
Media móvil 30 días
ROWS BETWEEN 29 PRECEDING AND CURRENT ROW. Tendencia mensual.
CEN
Media centrada
ROWS BETWEEN 3 PRECEDING AND 3 FOLLOWING. Ventana simétrica alrededor de la fila actual.
💡 También funciona con MIN, MAX: MAX OVER con una ventana deslizante te da el máximo de los últimos N días. Muy útil para detectar picos anómalos.
✅ Vs Python/pandas: df['ventas'].rolling(7).mean() ≡ AVG(ventas) OVER (ORDER BY fecha ROWS BETWEEN 6 PRECEDING AND CURRENT ROW). Directo en SQL, sin extraer datos.
SQL — Media móvil 7 y 30 días
SELECT
  fecha,
  ventas_diarias,

  -- Media móvil 7 días
ROUND(AVG(ventas_diarias) OVER ( ORDER BY fecha ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ), 0) AS mm_7d,
  -- Media móvil 30 días
ROUND(AVG(ventas_diarias) OVER ( ORDER BY fecha ROWS BETWEEN 29 PRECEDING AND CURRENT ROW ), 0) AS mm_30d,
  -- Máximo de los últimos 7 días
MAX(ventas_diarias) OVER ( ORDER BY fecha ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS max_7d
FROM ventas_diarias
ORDER BY fecha

09 · Segmentación

NTILE()
— Segmentación RFM

Divide clientes en N grupos de igual tamaño. La base del análisis RFM en SQL puro.

NTILE(4)
Cuartiles. Q4 = top 25% (los mejores). Q1 = bottom 25%.
→ RFM Monetario
NTILE(10)
Deciles. D10 = top 10% de clientes por valor.
→ Segmentación premium/mass
RFM combinado
CONCAT(r_score, f_score, m_score) = segmento 3D
→ Champions (444), At Risk (244), Lost (111)
⚠️ Grupos desiguales: Si el número de filas no es divisible exactamente entre N, NTILE reparte las filas extra empezando por el primer grupo. No todos los grupos son exactamente del mismo tamaño.
SQL — Análisis RFM completo
WITH metricas_cliente AS (
  SELECT
    cliente_id,
    MAX(fecha_pedido) AS ultima_compra,
    COUNT(*) AS num_pedidos,
    SUM(importe) AS gasto_total
  FROM pedidos
  GROUP BY cliente_id
),
rfm_scores AS (
  SELECT cliente_id,
-- Recencia: compra reciente = score alto NTILE(4) OVER (ORDER BY ultima_compra) AS r, -- Frecuencia NTILE(4) OVER (ORDER BY num_pedidos) AS f, -- Monetario NTILE(4) OVER (ORDER BY gasto_total) AS m   FROM metricas_cliente
)
SELECT *,
  CASE
    WHEN r=4 AND f=4 AND m=4 THEN 'Champions'
    WHEN r>=3 AND m>=3 THEN 'Loyal'
    WHEN r<=2 AND m>=3 THEN 'At Risk'
    ELSE 'Needs Attention'
  END AS segmento
FROM rfm_scores
10 · Valores extremos

FIRST_VALUE &
LAST_VALUE

Primer y último valor de la ventana. Imprescindibles para variaciones respecto a base.

FIRST_VALUE
Primer valor de la partición según el ORDER BY. Sin problemas de frame.
→ Precio inicial, primer pedido, baseline
LAST_VALUE ⚠️
Último valor de la partición. SIEMPRE especifica el frame completo.
→ Precio actual, estado final
❌ Bug clásico de LAST_VALUE: Sin frame explícito devuelve la fila actual (inútil). Siempre añade ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING.
💡 Alternativa a LAST_VALUE: En muchos casos es más legible usar FIRST_VALUE con ORDER BY DESC para obtener el último valor sin preocuparte por el frame.
SQL — Variación precio respecto a inicio
SELECT
  producto_id, mes, precio,

  -- Precio del primer mes (base de comparación)
FIRST_VALUE(precio) OVER ( PARTITION BY producto_id ORDER BY mes ) AS precio_inicial,
  -- Precio más reciente (FRAME obligatorio)
LAST_VALUE(precio) OVER ( PARTITION BY producto_id ORDER BY mes ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING -- ← OBLIGATORIO ) AS precio_actual,
  -- Variación % respecto al precio inicial
ROUND((precio - FIRST_VALUE(precio) OVER ( PARTITION BY producto_id ORDER BY mes )) * 100.0 / NULLIF(FIRST_VALUE(precio) OVER ( PARTITION BY producto_id ORDER BY mes ), 0), 2) AS var_pct
FROM precios_historicos ORDER BY producto_id, mes
11 · Distribución

PERCENT_RANK
& CUME_DIST

Percentiles y distribución acumulada directamente en SQL. Sin Python, sin exportar datos.

PERCENT_RANK
Rango relativo de la fila. Valor entre 0 y 1. Primera fila = 0 siempre.
(rank - 1) / (total_filas - 1)
→ ¿En qué percentil está este cliente?
CUME_DIST
Proporción de filas con valor ≤ al actual. Nunca devuelve 0.
filas_menores_o_igual / total_filas
→ ¿Qué % de pedidos cuesta menos que este?
💡 Uso práctico: Filtra WHERE PERCENT_RANK() OVER (...) >= 0.9 para quedarte con el top 10% de clientes por gasto. Mucho más elegante que un NTILE(10) WHERE n=10.
SQL — Percentil de clientes
SELECT
  cliente_id, gasto_total,

  -- Percentil del cliente en la distribución de gasto
ROUND(PERCENT_RANK() OVER ( ORDER BY gasto_total ) * 100, 1) AS percentil,
  -- Distribución acumulada
ROUND(CUME_DIST() OVER ( ORDER BY gasto_total ) * 100, 1) AS cume_dist_pct,
  -- Etiqueta de segmento por percentil
CASE WHEN PERCENT_RANK() OVER (ORDER BY gasto_total) >= .9 THEN 'Top 10% — Premium' WHEN PERCENT_RANK() OVER (ORDER BY gasto_total) >= .5 THEN '50-90% — Standard' ELSE 'Bottom 50% — Basic' END AS segmento
FROM metricas_cliente
12 · Caso real

Análisis de
cohortes con SQL

Retención mes a mes por cohorte de primera compra. El análisis más importante en SaaS y e-commerce.

①
Fecha de primera compra por cliente
MIN(fecha) OVER (PARTITION BY cliente_id) — mes de la cohorte.
②
Período dentro de la cohorte
Diferencia en meses entre fecha actual y fecha primera compra. 0=mes 1, 1=mes 2...
③
Tasa de retención
Clientes en período N / Clientes en período 0. Expresado en %.
CohorteM0M1M2M3
Ene-25100%62%48%41%
Feb-25100%58%44%—
Mar-25100%65%——
SQL — Cohortes de retención
WITH primera_compra AS (
  SELECT cliente_id, fecha_pedido,
MIN(fecha_pedido) OVER ( PARTITION BY cliente_id ) AS fecha_primera_compra   FROM pedidos
),
con_cohorte AS (
  SELECT
    cliente_id,
    DATE_TRUNC('month', fecha_primera_compra)
      AS mes_cohorte,
DATEDIFF(month, fecha_primera_compra, fecha_pedido) AS periodo_cohorte   FROM primera_compra
)
SELECT
  mes_cohorte,
  periodo_cohorte,
  COUNT(DISTINCT cliente_id) AS clientes,
ROUND(COUNT(DISTINCT cliente_id) * 100.0 / FIRST_VALUE(COUNT(DISTINCT cliente_id)) OVER (PARTITION BY mes_cohorte ORDER BY periodo_cohorte),1) AS retencion_pct FROM con_cohorte
GROUP BY mes_cohorte, periodo_cohorte
ORDER BY mes_cohorte, periodo_cohorte
13 · Debugging

Los 5 errores más comunes con Window Functions

❌ 1. LAST_VALUE sin frame

Sin ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING, devuelve la fila actual. Siempre especifica el frame completo.
❌ 2. ORDER BY texto en fechas

ORDER BY mes_nombre (texto) → orden alfabético, no cronológico. Siempre ordena por columna numérica de mes o por columna DATE.
❌ 3. Window function en WHERE

WHERE ROW_NUMBER() OVER(...) = 1 → error. Envuelve en CTE y filtra en el SELECT exterior.
❌ 4. PARTITION BY olvidado

Sin PARTITION BY, la ventana es toda la tabla. ROW_NUMBER va del 1 al total, sin reiniciarse por cliente o departamento.
❌ 5. ROWS vs RANGE

RANGE puede incluir filas extra con el mismo valor. Para acumulados y medias móviles usa siempre ROWS BETWEEN, no RANGE BETWEEN.
✅ Regla de oro

Prueba siempre tus window functions con un dataset pequeño y datos conocidos antes de ejecutar en producción. Los bugs silenciosos son los más peligrosos.
14 · Performance

Optimización
de Window Functions

①
Filtra antes con CTE
Reduce filas antes de calcular la ventana. WHERE en CTE previo, no después.
②
Agrupa funciones con mismo OVER
Varias funciones con idéntico OVER() en un mismo SELECT → una sola pasada por los datos.
③
PARTITION BY = columnas de partición de la tabla
En BigQuery y Snowflake, usa las mismas columnas de partición/clustering en el PARTITION BY.
💡 BigQuery: Usa QUALIFY en vez de CTE para filtrar resultados de window functions. Es más eficiente y evita una subquery extra. QUALIFY ROW_NUMBER() OVER(...) = 1
❌ Lento — múltiples pasadas
SELECT id, val,
  SUM(val) OVER (PARTITION BY dep ORDER BY fecha
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) ytd,
  AVG(val) OVER (PARTITION BY dep ORDER BY fecha
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) mm7
FROM (   -- Subquery innecesaria: los motores no pueden
  -- fusionar las dos ventanas si están separadas
  SELECT * FROM tabla WHERE fecha > '2025-01-01'
) t
✅ Rápido — filtra primero, agrupa ventanas
WITH filtrado AS (
  SELECT * FROM tabla
  WHERE fecha > '2025-01-01' -- ← filtra PRIMERO
)
SELECT id, dep, fecha, val,
-- Mismo OVER → una sola pasada SUM(val) OVER w AS ytd, AVG(val) OVER w AS media_acum, MAX(val) OVER w AS max_acum FROM filtrado
WINDOW w AS (
  PARTITION BY dep ORDER BY fecha
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)
15 · Sintaxis avanzada

Cláusula WINDOW
— Alias de ventana

Define la ventana una sola vez y reutilízala. Código más limpio y motores más eficientes.

✅
Una definición, múltiples funciones
WINDOW w AS (PARTITION BY ... ORDER BY ...) → todas las funciones usan OVER w.
✅
Mantenimiento simplificado
Cambias la partición en un sitio. Sin riesgo de inconsistencias entre funciones.
🔧 Compatibilidad: PostgreSQL ✅ BigQuery ✅ Snowflake ✅ SQL Server 2022+ ✅ MySQL 8+ ✅ SQLite 3.25+ ✅
⚠️ SQL Server < 2022: La cláusula WINDOW no está disponible. Repite el OVER completo en cada función o usa CTEs intermedias.
SQL — WINDOW alias (PostgreSQL / BigQuery)
SELECT
  vendedor_id, region, mes, ventas,

  -- Todas usan el mismo alias de ventana
ROW_NUMBER() OVER w_region AS ranking, SUM(ventas) OVER w_region AS acum_region, AVG(ventas) OVER w_region AS media_acum, LAG(ventas) OVER w_region AS ventas_mes_ant
FROM ventas_mensuales

-- Definición única al final de la query
WINDOW w_region AS ( PARTITION BY region ORDER BY mes ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW )
-- SQL Server: sin cláusula WINDOW, repite el OVER
-- ROW_NUMBER() OVER (PARTITION BY region ORDER BY mes)
16 · Compatibilidad

Diferencias entre BigQuery · Snowflake · PostgreSQL · SQL Server

BigQuery
✅ QUALIFY — Filtra window functions directamente sin CTE
✅ DATE_DIFF(fecha2, fecha1, unit)
✅ WINDOW alias — soportado
💡 PARTITION BY debe coincidir con columnas de partición de la tabla para evitar full scans
Snowflake
✅ QUALIFY — Igual que BigQuery
✅ DATEDIFF(unit, fecha1, fecha2)
✅ WINDOW alias — soportado
💡 Usa columnas de clustering en PARTITION BY para mejor rendimiento
PostgreSQL
❌ QUALIFY — No existe. Usa CTE
✅ AGE / fecha2 - fecha1
✅ WINDOW alias — soportado
✅ FILTER(WHERE ...) dentro de funciones de agregado con OVER
SQL Server
❌ QUALIFY — No existe. Usa CTE
✅ DATEDIFF(unit, fecha1, fecha2)
⚠️ WINDOW alias — Solo desde 2022
⚠️ ROWS/RANGE con UNBOUNDED limitado en versiones < 2012
17 · BigQuery / Snowflake

QUALIFY
— Filtrar sin CTE

La cláusula que simplifica el código al eliminar CTEs de deduplicación.

❌ SQL estándar: Window function en WHERE → error. Obligatorio usar CTE o subquery exterior.
✅ QUALIFY: Filtra window functions directamente. Misma potencia, menos código. BigQuery y Snowflake.
💡 Usos clave:
• Deduplicar: QUALIFY ROW_NUMBER() OVER(...) = 1
• Top N por grupo: QUALIFY RANK() OVER(...) <= 5
• Último evento: QUALIFY ROW_NUMBER() OVER(ORDER BY ts DESC) = 1
❌ No disponible en: PostgreSQL, SQL Server, MySQL. Usa CTE en estos motores.
❌ SQL estándar — requiere CTE
WITH ranked AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY cliente_id
      ORDER BY fecha_pedido DESC
    ) AS rn
  FROM pedidos
)
SELECT * FROM ranked WHERE rn = 1
✅ BigQuery / Snowflake — QUALIFY
SELECT *
FROM pedidos
QUALIFY ROW_NUMBER() OVER ( PARTITION BY cliente_id ORDER BY fecha_pedido DESC ) = 1
-- Top 3 por región con QUALIFY

QUALIFY DENSE_RANK() OVER ( PARTITION BY region ORDER BY ventas DESC ) <= 3

Cheatsheet completo — Window Functions

ROW_NUMBER()
Numeración única
Sin empates. 1,2,3,4...
→ Deduplicar, paginar
RANK()
Ranking con saltos
Empates = mismo nº + salta. 1,1,3,4...
→ Competiciones, concursos
DENSE_RANK()
Ranking sin saltos
Empates = mismo nº sin saltar. 1,1,2,3...
→ Top N por grupo
LAG(col, n, def)
N filas atrás
Valor de la fila N posiciones anterior
→ MoM, YoY, variaciones
LEAD(col, n, def)
N filas adelante
Valor de la fila N posiciones posterior
→ Churn, duración, siguiente estado
SUM() OVER
Suma / acumulado
YTD con UNBOUNDED PRECEDING
→ Running total, porcentajes
AVG() OVER
Media móvil
N PRECEDING → N días móviles
→ Suavizar series temporales
NTILE(n)
N grupos iguales
Divide partición en N cubos
→ RFM, cuartiles, deciles
PERCENT_RANK()
Percentil 0-1
Rango relativo en la distribución
→ Top 10%, bottom 20%
CUME_DIST()
Distribución acumulada
% de filas con valor ≤ al actual
→ Análisis de distribución
FIRST_VALUE(col)
Primer valor
Sin problemas de frame
→ Baseline, precio inicial
LAST_VALUE(col) ⚠️
Último valor
SIEMPRE frame UNBOUNDED
→ Precio actual, estado final
